Git / GitHubコードのバージョン管理と共有

たとえ話
Git = 作業の「セーブ機能」。RPGのセーブポイントのように、いつでも過去の状態に戻れます。
GitHub = セーブデータを保存する「クラウド上のセーブスロット」。他の人とも共有できます。
用語意味たとえ
リポジトリ プロジェクト全体の保管場所 1つのゲームのセーブフォルダ
コミット 変更を記録すること セーブする
プッシュ 自分のPCのセーブをGitHubに送る セーブデータをクラウドにアップロード
プル GitHubの最新をPCにダウンロード クラウドから最新セーブを取得
ブランチ メインとは別の作業場所を作る 下書き用のコピーを作って、完成したら本物に合体
クローン GitHubからプロジェクトを丸ごとコピー 友達のセーブデータを自分のPCにダウンロード

よく使うコマンド

# 変更の確認（今どのファイルが変わったか見る）
git status

# ファイルをセーブ対象に追加
git add ファイル名

# セーブする（メッセージ付き）
git commit -m "ログイン機能を追加"

# GitHubに送る
git push

# GitHubから最新を取得
git pull

ターミナルコマンドライン / シェル

たとえ話
パソコンへの「文字による指示書」。普段はマウスでクリックして操作しますが、ターミナルでは文字を打って指示を出します。 プログラマーにとっては、マウスより速くて正確なのでよく使います。
あなた ──「このフォルダのファイル一覧を見せて」──→ ターミナル ──→ パソコン ↓ あなた ←── ファイル一覧が表示される ←──────────────────────── 結果を返す

よく使うコマンド

# 今いるフォルダを確認
pwd

# ファイル一覧を表示
ls

# フォルダを移動
cd フォルダ名

# フォルダを作る
mkdir フォルダ名

# ファイルの中身を見る
cat ファイル名

npmNode Package Manager

たとえ話
「プログラミング用のApp Store＋自動インストーラー」。 他の人が作った便利な部品（パッケージ）を、コマンド1つでダウンロードして自分のプロジェクトで使えます。
あなたのアプリ ├── 自分で書いたコード └── npm で取り寄せた「部品」たち ├── next → Webアプリの土台 ├── react → 画面を作る仕組み ├── prisma → データベースとやり取りする道具 ├── tailwindcss → デザインを簡単にするCSS └── ...他にも数百個

重要なファイル

ファイル役割たとえ
package.json 必要なパッケージのリスト 料理のレシピ（材料一覧）
package-lock.json バージョンの正確な記録 材料の産地・ロット番号まで記録した仕入れ台帳
node_modules/ ダウンロードされたパッケージ本体 実際に届いた材料（非常に大きいのでGitHubには上げない）

よく使うコマンド

# パッケージを一括インストール（材料を全部そろえる）
npm install

# 開発サーバーを起動（手元で動かして確認する）
npm run dev

# ビルド（本番用に最適化して組み立てる）
npm run build

# テスト実行（正しく動くか確認する）
npm run test

# 新しいパッケージを追加
npm install パッケージ名

.env ファイル環境変数ファイル（ドットイーエヌブイ）

たとえ話
アプリを「お店」に例えると、.env は「金庫の暗証番号や鍵が書かれたメモ」。 お店の設計図（コード）は誰に見せてもOKですが、暗証番号は店長だけが持つべきものです。

中に書く内容の例

# データベースの接続先（住所とパスワード）
DATABASE_URL="postgresql://user:password@localhost:5432/mydb"

# 外部サービスのAPIキー（秘密の通行証）
PAYJP_SECRET_KEY="sk_test_xxxxxxxxxxxxx"

# ログイン認証用の暗号化キー
JWT_SECRET="ランダムな長い文字列"

# アプリのURL
NEXT_PUBLIC_APP_URL="http://localhost:3000"
よくある項目用途
データベース接続情報DBのホスト名・パスワード
APIキー決済・地図・メール送信などの外部サービス
認証用シークレットログイン機能の暗号化
アプリのURL開発用(localhost)と本番用で切り替え
動作モード開発モード / 本番モードの切り替え
ポイント
.env は各環境（自分のPC、本番サーバーなど）ごとに作るファイルです。GitHubには上げず、代わりに .env.example（中身が空のテンプレート）を共有します。

.vscode/

フロントエンド / バックエンド画面側 / サーバー側

たとえ話
レストランで例えると――
フロントエンド = ホール（お客さんが見る場所：メニュー、テーブル、内装）
バックエンド = 厨房（お客さんから見えない場所：調理、在庫管理、会計処理）
ユーザー（お客さん） │ │ ブラウザで画面を見る・ボタンを押す ↓ ┌──────────────────────────────────┐ │ フロントエンド（ホール） │ │ HTML / CSS / JavaScript │ │ React, Vue, Tailwind CSS など │ └──────────────┬───────────────────┘ │ データをください！（APIリクエスト） ↓ ┌──────────────────────────────────┐ │ バックエンド（厨房） │ │ Node.js, Python, Ruby など │ │ データベースとやり取り │ │ ログイン処理、決済処理 │ └──────────────────────────────────┘
ポイント
Next.js のようなフレームワークは、フロントエンドとバックエンドを1つのプロジェクトで両方できるのが特徴です（フルスタックフレームワークと呼びます）。

APIApplication Programming Interface

たとえ話
レストランの「注文窓口」。 お客さん（フロントエンド）が「カレーください」と注文すると、厨房（バックエンド）が調理して「はいどうぞ」と返してくれる。 この注文の受け渡しルールがAPIです。
① フロントエンドが注文を出す 「/api/users のデータをください」（リクエスト） │ ↓ ② バックエンドが処理する データベースからユーザー情報を取得 │ ↓ ③ 結果を返す { "name": "田中", "age": 30 }（レスポンス）

2種類のAPI

種類説明例
自分のAPI 自分のアプリ内でフロントとバックをつなぐ /api/auth/login（ログイン処理）
外部API 他社のサービスを借りて使う Pay.jp（決済）、Google Maps（地図）

よく使うデータベース

名前特徴

ORM（オーアールエム）とは？

たとえ話
データベースとの「通訳」。データベースには専用の言語（SQL）で話しかける必要がありますが、ORMを使えば普段のプログラミング言語で操作できます。 
// ORMなし（SQL言語で直接書く）
SELECT * FROM users WHERE age > 20;

// ORMあり（Prismaの場合：普通のコードで書ける）
const users = await prisma.user.findMany({
  where: { age: { gt: 20 } }
});

フレームワーク / ライブラリ開発を助ける道具セット

たとえ話
ライブラリ = 工具箱。必要な工具（ドライバー、ペンチ等）を自分で選んで使う。
フレームワーク = 住宅キット。家の骨組みがすでにできていて、壁紙や家具を選ぶだけで家が建つ。
ライブラリフレームワーク

デプロイアプリを公開すること

たとえ話
お店の「開店作業」。自宅のキッチンで試作（開発）したものを、実際のお店（サーバー）に持っていってお客さんに提供できるようにすること。
開発（自分のPC） 本番（サーバー） ┌───────────────┐ ┌───────────────┐ │ コードを書く │ │ ユーザーが │ │ テストする │── git push ─→ GitHub ──→│ アクセスできる│ │ 動作確認する │ （自動デプロイ）│ 状態にする │ └───────────────┘ └───────────────┘ localhost:3000 example.com

CI/CD（シーアイ・シーディー）とは？

たとえ話
「自動お届けサービス」。GitHubにコードを送る（push）だけで、テスト → ビルド → デプロイ（公開）まで全自動でやってくれます。手作業でサーバーにファイルをコピーする必要がなくなります。

サービス特徴

GCPGoogle のクラウド。大規模サービス向き
AWSAmazon のクラウド。世界シェアNo.1
VPS自分専用のサーバーを借りる。自由度が高い

Dockerドッカー

たとえ話
「引っ越し用の段ボール箱」。アプリと、動かすために必要なもの（OS設定・ライブラリ等）を全部1つの箱（コンテナ）に詰めて、どのPCでもそのまま動かせるようにする仕組み。
問題: 「自分のPCでは動くのに、他の人のPCでは動かない！」 原因: PCごとにインストールされているソフトのバージョンが違う ┌────────────────┐ ┌────────────────┐ │ Aさんの PC │ │ Bさんの PC │ │ Node.js 18 │ │ Node.js 16 │ ← バージョンが違う！ │ PostgreSQL 15 │ │ PostgreSQL 13 │ └────────────────┘ └────────────────┘ 解決: Docker を使うと全員同じ環境で動く ┌──── Docker コンテナ ────┐ │ Node.js 18 │ ← バージョン固定 │ PostgreSQL 15 │ │ アプリのコード │ │ 必要な設定すべて │ └─────────────────────────┘ どのPCでもこの箱をそのまま動かすだけ！

重要なファイル

ファイル役割たとえ
Dockerfile コンテナの設計図 「箱の中にこれを入れてね」という指示書
docker-compose.yml 複数コンテナをまとめて管理 「アプリの箱」と「DBの箱」をセットで起動する指示書
